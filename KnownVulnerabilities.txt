1. SQL Injection - Login Email
   Endpoint: POST /auth/login
   Parameter: email (JSON body)
   File: backend/api/models/User.php -> authenticateDirect()
   Description: The email value is directly concatenated into a SQL query
   without parameterized statements. An attacker can inject SQL via the
   email field to bypass authentication or extract data.
   Vulnerable Code: backend/api/models/User.php line 41
     "SELECT * FROM users WHERE email = '$email' AND password_hash = '$password'"
   Fix: Use parameterized queries with bound parameters:
     $stmt = $db->prepare("SELECT * FROM users WHERE email = :email");
     $stmt->bindValue(':email', $email, SQLITE3_TEXT);
   Example: email = ' OR 1=1 --

2. SQL Injection - Wine Detail (ID)
   Endpoint: GET /wines/:id
   Parameter: id (URL path)
   File: backend/api/models/Wine.php -> getByIdDirect()
   Description: The wine ID is directly concatenated into a SQL query.
   The route accepts non-numeric values, allowing SQL injection.
   Vulnerable Code: backend/api/models/Wine.php line 89

11. JWT "none" Algorithm Accepted (Broken Authentication)
    CWE: CWE-345 (Insufficient Verification of Data Authenticity)
    OWASP API: API2:2023 Broken Authentication
    Endpoint: Any authenticated endpoint
    Header: Authorization: Bearer <token>
    File: backend/api/config/jwt.php -> decode()
    Description: The JWT decoder checks the "alg" field in the token
    header. If alg is "none", it skips signature verification entirely
    and accepts the token payload as valid. An attacker can forge tokens
    by setting alg=none and providing arbitrary payload data.
    Vulnerable Code: backend/api/config/jwt.php lines 34-40
      if (strtolower($headerData['alg']) === 'none') {
          $data = json_decode(self::base64UrlDecode($payload), true);
          ...
          return $data;  // accepts token without any signature
      }
    Fix: Reject any token where alg is not the expected algorithm:
      if ($headerData['alg'] !== 'HS256') { return null; }
    Example: Header={"alg":"none","typ":"JWT"}, Payload={"user_id":1,"email":"admin@example.com","exp":9999999999}

12. JWT Signature Not Verified (Broken Authentication)
    CWE: CWE-347 (Improper Verification of Cryptographic Signature)
    OWASP API: API2:2023 Broken Authentication
    Endpoint: Any authenticated endpoint
    Header: Authorization: Bearer <token>
    File: backend/api/config/jwt.php -> decode()
    Description: Even for HS256 tokens, when the signature doesn't match,
    the code only logs a warning but still accepts the token. An attacker
    can modify the payload of any JWT token (e.g., change user_id) without
    needing to know the secret key.
    Vulnerable Code: backend/api/config/jwt.php lines 46-48
      if ($signature !== $validSignature) {
          error_log("JWT signature mismatch for token, but accepting anyway");
      }
    Fix: Return null (reject the token) when the signature does not match:
      if ($signature !== $validSignature) { return null; }

13. Directory Listing
    URL: https://taintedport.com/files/
    URL: https://api.taintedport.com/files/
    File: docker/nginx.conf
    Description: Nginx is configured with autoindex enabled on /files/,
    which maps to the backend source directory. This exposes all PHP
    source code, the SQLite database file, and configuration files
    including the JWT secret. An attacker can download the database
    and extract user credentials.
    Vulnerable Code: docker/nginx.conf lines 47 and 79
      location /files/ {
          alias /var/www/backend/;
          autoindex on;
      }
    Fix: Remove the autoindex directive or set it to off:
      autoindex off;
    Or remove the /files/ location block entirely.

16. Missing Security Headers
    CWE: CWE-693 (Protection Mechanism Failure)
    Severity: LOW
    Endpoint: All endpoints
    Description: The application does not set critical security headers:
    - Strict-Transport-Security (HSTS) is missing, allowing downgrade attacks
    - X-Content-Type-Options: nosniff is missing, allowing MIME-type sniffing
      which can amplify XSS vulnerabilities
    These are commonly flagged by DAST scanners as security misconfigurations.
    Vulnerable Code: docker/nginx.conf (entire http block) and
      backend/api/index.php lines 4-11 (only sets CORS and Content-Type)
    Fix: Add security headers in nginx.conf or backend:
      add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
      add_header X-Content-Type-Options "nosniff" always;
      add_header X-Frame-Options "DENY" always;

29. Reflected XSS - Contact Form Preview (Server-Side)
    CWE: CWE-79 (Improper Neutralization of Input During Web Page Generation)
    Severity: MEDIUM
    Endpoint: POST /contact/preview
    Parameters: name, email, subject, message (form body)
    File: backend/api/controllers/ContactController.php -> preview()
    Description: The contact form preview endpoint receives form data via
    a standard HTML form POST (application/x-www-form-urlencoded) and renders
    an HTML confirmation page with the form values embedded directly in the
    HTML without escaping. Unlike the other XSS vulnerabilities in this app
    (which are DOM-based via dangerouslySetInnerHTML), this is a traditional
    server-side reflected XSS: the PHP backend generates HTML with unsanitized
    user input, and the browser renders it as part of the page. All four form
    fields (name, email, subject, message) are vulnerable.
    Vulnerable Code: backend/api/controllers/ContactController.php
      $name = isset($_POST['name']) ? $_POST['name'] : '';
      ...
      echo "<div class=\"value\">$name</div>";
      (form values interpolated directly into HTML without htmlspecialchars)
    Fix: HTML-encode all user input before embedding in the page:
      $name = htmlspecialchars($_POST['name'] ?? '', ENT_QUOTES, 'UTF-8');
      $email = htmlspecialchars($_POST['email'] ?? '', ENT_QUOTES, 'UTF-8');
    Example: name=<script>alert(document.cookie)</script>

23. Privilege Escalation via JWT Claim Forgery (Vulnerability Chain)
    CWE: CWE-269 (Improper Privilege Management), CWE-345 (Insufficient Verification of Data Authenticity)
    Severity: CRITICAL
    Endpoint: All /admin/* endpoints
    File: backend/api/controllers/AdminController.php -> requireAdmin()
    File: backend/api/config/jwt.php -> decode()
    Description: The admin authorization check reads the is_admin flag from
    the JWT token payload instead of the database. Combined with the existing
    JWT vulnerabilities (#11 "none" algorithm or #12 signature bypass), an
    attacker can forge a JWT token with is_admin=true to gain admin access.
    Vulnerable Code: backend/api/controllers/AdminController.php line 18
      if (empty($authUser['is_admin'])) { ... }
      ($authUser comes from JWT decode, not from database lookup)

    is_admin to true, (4) Re-encode with alg=none or any signature,
    (5) Use the forged token to access /admin/* endpoints.

27. SSRF via Wine Import URL -> JWT Secret -> Admin Token Forgery (Vulnerability Chain)
    CWE: CWE-918 (Server-Side Request Forgery)
    Severity: HIGH
    Endpoint: POST /wines/import-url
    Parameter: url (JSON body)
    File: backend/api/controllers/WineController.php -> importFromUrl()
    Description: The wine import endpoint fetches content from a user-supplied
    URL using file_get_contents(), which supports file://, http://, and other
    PHP stream wrappers. An attacker can read arbitrary local files by
    providing a file:// URL. This enables a multi-step attack chain:
    (1) Use SSRF to read file:///var/www/backend/api/config/jwt.php
    (2) Extract the hardcoded JWT secret from the PHP source
    (3) Forge a properly HS256-signed JWT with is_admin=true
    (4) Access all /admin/* endpoints with a cryptographically valid token
    Unlike chain #23 (which uses alg=none or signature bypass), this chain
    produces a properly signed token that survives fixing those JWT bugs.
    Vulnerable Code: backend/api/controllers/WineController.php
      $content = @file_get_contents($data['url']);
      (no URL scheme validation, no SSRF protection)
    Fix: Validate the URL scheme (allow only https://), block private/internal
    IPs, and use curl with CURLOPT_PROTOCOLS instead of file_get_contents():
      $parsed = parse_url($data['url']);
      if (!in_array($parsed['scheme'], ['http', 'https'])) { reject; }
    Chain Dependencies: #11/#12 are NOT required — this chain uses the real

    Example: {"url": "file:///var/www/backend/api/config/jwt.php"}

28. SQL Injection -> TOTP Secret Extraction -> 2FA Bypass -> Account Takeover (Vulnerability Chain)
    CWE: CWE-89 (SQL Injection), CWE-308 (Use of Single-factor Authentication)
    Severity: CRITICAL
    Endpoints: GET /wines?search= (step 1), POST /auth/login (step 4)
    Description: This chain combines SQL injection with TOTP secret theft to
    fully bypass two-factor authentication and take over any account,
    including admin. The attack proceeds in four steps:
    (1) Use SQL injection (#3 wine search, or #2/#4) to extract email,
        password_hash, and totp_secret for all users from the users table
    (2) Crack the password hash offline (users have weak passwords)
    (3) Use the stolen totp_secret to compute valid TOTP codes
    (4) Login as the victim with email + cracked password + valid TOTP code
    This demonstrates that 2FA provides no additional security when the TOTP
    secret is stored alongside credentials that are already extractable via
    SQL injection. The entire second factor is nullified.
    Chain Dependencies: Requires #3 (or #2/#4) for SQL injection.
    Example SQLi payload (wine search, 8 columns):
      search=' UNION SELECT 1,email,password_hash,name,5,6,totp_secret,totp_enabled FROM users--
    Example SQLi payload (wine reviews, 5 columns):
      GET /wines/0 UNION SELECT 1,email,password_hash,totp_secret,totp_enabled FROM users--/reviews
